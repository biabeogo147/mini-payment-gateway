\begin{longtable}{@{}L{0.27\textwidth}L{0.63\textwidth}@{}}

Base path & All documented endpoints use the \code{/v1} prefix. \\
\bodyrowrule
Payload format & Requests and responses are JSON, except for outbound webhooks
that the gateway delivers to the merchant's own endpoint. \\
\bodyrowrule
Timestamps & Time values use UTC ISO 8601 format. \\
\bodyrowrule
Amounts & Monetary values are decimal strings so clients do not depend on
binary floating-point behavior. \\
\bodyrowrule
Idempotency & \code{X-Idempotency-Key} is optional for merchant requests that
may be retried after network uncertainty. Business-level duplicate handling is
still enforced even when this header is omitted. \\
\bodyrowrule
Error envelope & All error responses return an error code, a human-readable
message, an optional request identifier, and structured details when useful. \\
\bottomrule
\end{longtable}

\apiblockheading{Merchant HMAC headers.}
Merchant backend payment and refund requests require the following headers:
\code{X-Merchant-Id}, \code{X-Access-Key}, \code{X-Timestamp}, and
\code{X-Signature}. When technical request replay protection is desired,
\code{X-Idempotency-Key} may also be included.

Canonical signing string:

\begin{lstlisting}[caption={Merchant HMAC signing string}]
{timestamp}.{method}.{path}.{body_sha256_hex}
\end{lstlisting}

Signature value:

\begin{lstlisting}[caption={Merchant HMAC signature}]
hex(hmac_sha256(secret_key, signing_string))
\end{lstlisting}

\apiblockheading{Shared error response shape.}
\begin{lstlisting}[language=json,caption={Error response example}]
{
  "error_code": "PAYMENT_ALREADY_SUCCESS",
  "message": "Payment already succeeded for this order.",
  "request_id": "req_01J...",
  "details": {}
}
\end{lstlisting}

The full error catalogue later in this guide explains what each important code
means, when retrying helps, and what the caller should do next.
